# 撰写排障笔记

从飞书对话、合并转发消息、或用户口头描述中，提取排障过程并撰写结构化 Obsidian 笔记。

## 前置步骤：加载规范 skill

每次写之前 MUST 先加载 `troubleshooting-doc` 获取文档结构和格式规范：
```
skill_view(name="troubleshooting-doc")
```

## 信息提取流程

### 来源：飞书合并转发消息

当用户提供飞书 `merge_forward` 消息内容时：

1. **解析时间线**：按 `[时间] 发送者:` 格式提取每个人发言的时间和内容
2. **识别参与者**：区分用户、IT 支持、系统消息等角色
3. **提取操作链**：谁在什么时间做了什么操作 → 结果如何
4. **识别关键发现**：用 `**发现**` 标记每次尝试后的结论
5. **判断状态**：是否给出了最终解决方案？是否确认生效？

### 来源：飞书群聊

1. 先用 `lark-cli im +chat-messages-list` 拉取消息
2. 需要 `--as user` 身份（bot 无法读 P2P）
3. 合并转发消息的 `content` 字段已包含完整文本

## 保存路径

```bash
~/Library/Mobile\ Documents/com~apple~CloudDocs/笔记/Obsidian\ Vault/Troubleshooting/<标题>.md
```

## 语言规则

- 正文：中文
- 技术术语、命令、文件路径、环境变量：保留英文原样
- 不要翻译命令输出内容

## 写作规则

1. **禁止编造**：所有命令输出、配置、时间必须来自原始消息，不得虚构
2. **区分已解决/未解决**：
   - 已解决：有 `## 根因` + `## 解决方案` 章节
   - 未解决：用 `## 根因分析` + `## 待尝试方案`，第一行写 `**状态：未解决**`
3. **表格优先**：多步骤对比用表格替代纯文本描述
4. **代码块标注语言**：bash / text / json / 不标注
5. **高亮发现**：每个阶段的结论用 `**发现**` 粗体
6. **标注信息源**：文档末尾注明数据来自哪个飞书 chat_id / message_id

## 完整的撰写步骤

### Step 1：加载规范
```python
skill_view(name="troubleshooting-doc")
```

### Step 2：提取信息
从用户提供的对话中提取：
- 用户信息、时间范围、设备环境
- 每次操作 → 结果的因果链
- 最终状态（解决 / 未解决 / 临时绕过）

### Step 3：判断文档类型
- 网络设备类（交换机/AP/防火墙）→ 按 troubleshooting-doc 完整格式
- 客户端/软件类（macOS/Node.js/VPN 等）→ 用本 skill 简化模板

### Step 4：写出并验证
- write_file 到 Obsidian 路径
- read_file 确认开头 10 行
- 告知用户完整路径

## 常见错误

| 错误 | 纠正 |
|------|------|
| 忘记加载 troubleshooting-doc | Step 1 必须执行 |
| 把用户猜测当结论 | 区分「用户操作事实」vs「用户猜测」，猜测标为「待验证」 |
| 编造不存在的路径/命令 | 所有内容必须来自原始消息 |
| 用英文写正文 | 正文中文，术语保留英文 |
| 遗漏信息源 | 必须注明飞书 chat_id/message_id |
